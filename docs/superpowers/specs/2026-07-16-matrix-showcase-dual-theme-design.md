# 电商工作台重建 · 样板间子项目设计文档

- **日期**：2026-07-16
- **状态**：已实施（阶段一完成，含最终整体 review 补的一处修正，见文末「实施后补记」）
- **子项目定位**：整站重建的第一阶段（共四阶段，见下方「整体路线图」），本子项目只覆盖「顶栏共享外壳 + 价格带沙盘页」的结构与双主题改造

## 背景

当前项目要做一次较大的前端重建，用户提出四点诉求：

1. 升级页面结构和交互逻辑，让信息架构更合理
2. 视觉重构——当前偏"传统 AI 化"模板感，想要更有质感的前端效果
3. 新增深浅双主题切换，两套主题对应两种独立的页面风格（不是简单换色）
4. 竞品 3D 预览的视觉效果需要升级，但方向未定

体量横跨四个相对独立的关注点，不适合一次性重构，因此拆解为按依赖顺序推进的子项目，本文档只覆盖第一个。

### 已有的 `redesign/dark-theme` 分支

仓库里已存在一个 `redesign/dark-theme` 分支（本地 + origin 都有），从 `main` 分出后做过登录页 WebGL 场景重制、竞品3D预览从 ECharts-GL 换成原生 Three.js、全站深色令牌统一。用户已知晓这个分支，**但视觉方向不采用**——它只做了单一深色主题（没有真正的双主题体系），也没有触及本次要解决的结构/交互问题，因此这次重建**不基于该分支**，新分支从 `main` 切出（`main` 上有 `redesign/dark-theme` 没有的、更新的价格带沙盘功能提交）。`redesign/dark-theme` 分支保留不动，作为未来第 4 阶段（3D预览视觉专项）时是否复用其 Three.js vendor 文件（`three.module.min.js` 及配套的 orbitcontrols/后处理管线）的候选参考，届时再决定其最终处置（合并/归档/丢弃）。

## 整体路线图（四阶段，本文档只详细设计阶段一）

1. **样板间**（本文档）：顶栏/⋯菜单共享外壳 + 价格带沙盘页，验证新 IA 规则 + 双主题技术机制
2. **逐页铺开**：把阶段一验证过的规则复制到竞品对位、评论风向标、竞品3D预览（仅结构/主题适配，不含视觉专项升级）、报告管理、管理员面板、用户管理、设置
3. **3D 预览视觉专项**：竞品3D预览的可视化效果升级，具体方向届时再看demo决定（候选方向包括：数据可视化质感/游戏引擎质感/产品级质感，本次讨论中未拍板）
4. **收尾**：登录页是否需要独立的结构例外（WebGL 高光时刻）、整体验收

## 顶栏 + ⋯菜单信息架构调整

顶部 5 个 Tab（价格带沙盘/竞品对位/评论风向标/竞品3D预览/报告管理）结构合理，**不调整**。

问题在 ⋯ 菜单——现状是一份平铺列表（导出/导入/打印/变更日志/备份恢复/用户管理/设置/恢复出厂数据/退出登录），不区分受众和风险等级。调整为按分组呈现：

| 分组 | 内容 | 说明 |
|---|---|---|
| 个人偏好 | 功能显示设置 | 只影响自己（主题切换不放在这个菜单里，见下方「双主题技术机制」——它是顶栏右侧的常驻按钮，跟"开启编辑"同一视觉层级，因为高频、单击操作不适合藏进下拉菜单） |
| 账号 | 用户管理 | 双重身份：管理员管全员，普通成员只能改自己 PIN——所有人都能看到这一项，不是管理员专属 |
| 数据治理 | 导出 JSON 备份、从 JSON 恢复、打印/导出 PDF | 有实际影响但非破坏性，全员可用 |
| 管理员专区 | 变更日志、备份与恢复 | 视觉上单独隔开一块，非管理员不可见——`server.js` 里这两个接口都服务端强制 403（不是只在前端隐藏），修正了此前草稿里"变更日志全员可查看"的错误说法（那是逆向 PRD 文档描述 `redesign/dark-theme` 分支的说法，跟 `main` 分支现状不一致） |
| 危险操作 | 恢复出厂数据 | 前端隐藏（非破坏性强制），强化视觉警示（现有 `.danger` 样式基础上加强） |
| 会话 | 退出登录 | 独立于以上业务分组 |

## 侧栏三分区规则（跨页通用规范，本次先在沙盘页验证）

现状排查：五个页面的侧栏（`rail`）都是同一种模式——`rail-hint`（提示说明）、实际交互控件（编辑器/按钮/颜色选择器）、`stats`（只读统计数字）按 `rail-sec` 小标题平铺堆叠，三种性质不同的内容没有视觉区分，是全站通用问题，不是沙盘页独有——这也是选它做样板间的原因之一。

规则：
- **说明区**：视觉弱化（更小字号、静音色、图标前缀），不与操作控件抢注意力
- **操作区**：视觉主导，实际的编辑器/按钮/输入控件
- **统计区**：读数类内容独立成一条"卡片带"，不与操作控件混排

沙盘页侧栏现有内容（图例与分类编辑器、默认产品色、结构操作、批量调整说明、统计面板）按此规则重新分区，验证过的规则写入本文档供阶段二直接复用，不再重新讨论。

## 双主题技术机制

- `<html data-theme="dark|light">` 驱动，以 CSS 自定义属性为主要机制
- **共享 DOM 结构**：深浅两态使用同一份 HTML/组件骨架，不做双模板。纯装饰性元素（如深色态的极光渐变团）常驻 DOM，用 CSS/`data-theme` 控制显隐与强度，不算"结构分叉"
- **例外机制**：如果某个页面确实需要局部结构级差异（目前唯一候选是登录页，留到阶段四决定），单独开例外，不作为默认规则
- **偏好持久化**：沿用现有「功能显示设置」的模式——服务端按用户存储，换设备同步，而非仅 `localStorage`。具体存储位置（`settings.js` 扩展 / 新字段）在实施计划里定
- **默认值**：沿用现状深色为默认，浅色为可选切换；切换开关放顶栏右侧，与「开启编辑」同一视觉层级
- **未迁移页面的兼容处理**：阶段一完成时，只有共享外壳 + 沙盘页支持双主题，其余六个页面视觉仍是旧的纯深色实现。为避免用户切到浅色后其余页面"半新半旧"看起来破损，未迁移页面在阶段一期间**强制按深色渲染**，不跟随用户的浅色偏好，用一个简单的"该视图是否已适配主题"标记控制，等阶段二每个页面迁移完成后去掉各自的强制项

## 视觉语言

### 深色态 · 极光玻璃（Aurora Glass）

深邃底色 + 流动极光渐变团（缓慢漂移动效）+ 强玻璃拟态（`backdrop-filter` 模糊+饱和度提升）+ 发光描边。延续品牌薄荷绿 `#4ee0c1` 作为贯穿色，新增紫色系极光作为"焕然一新"的视觉爆点。

### 浅色态 · 商务瓷白

纯净留白 + 软彩色阴影（替代传统灰阴影）+ 精致细描边，克制但有质感，定位是"日常摸数据"场景下的低疲劳度界面。

### 通用约束（沿用现状，不因本次重建改变）

- `backdrop-filter` 只用于共享外壳层（顶栏/侧栏/下拉菜单），画布/板面内部不用——避免 `html2canvas` 导出 PNG 时的兼容性问题
- 动效遵循 `prefers-reduced-motion`：不满足时关闭极光漂移等装饰性动画
- 打印/PDF 走独立的 `@media print`，不受主题影响

## 验收标准（手工验收，本项目无自动化前端测试）

- 深浅两态在共享外壳 + 沙盘页视觉均完整可用，切换后无样式残留/闪烁
- 主题偏好换设备/重新登录后保持，与「功能显示设置」的持久化行为一致
- 沙盘页 PNG 导出在两种主题下都不受 `backdrop-filter` 影响，保真度不下降
- 未迁移的六个页面在浅色偏好下仍强制深色渲染，不出现半新半旧的视觉断层
- `prefers-reduced-motion` 开启时装饰性动效（极光漂移等）关闭
- 响应式断点（≤1080px / ≤760px）下侧栏三分区规则依然清晰可辨

## 实施后补记

8 个任务逐一实施+review 通过后，整体分支最终 review（不同于单任务 review，专门找跨任务才会暴露的问题）发现一处这份文档当初没覆盖到的缺口：

「未迁移的六个页面在浅色偏好下仍强制深色渲染」这条验收标准，实现时是靠 `.view[data-theme-ready="false"]` 这层选择器兜底的——但「变更日志」「备份与恢复」「用户管理」「功能显示设置」这几个浮层（`admin.js`/`users.js`/`settings.js` 管理的 `.sheet-mask`）不是 `.view` 的子元素，而是各自 `document.body.appendChild()` 挂到 body 下的，兜底层完全没覆盖到它们。浅色偏好下打开这几个浮层，除了「变更日志」（`.log-act.*` 徽标颜色硬编码深色值，在浅色浮层上直接变成一块块颜色不对的色块，是最明显的一处）之外，其余几个浮层是"整体变浅色但没设计过"，不算破损但也不算达标。

这不是某个任务实施错了，是设计阶段就没想到——「未迁移页面」当时只想到了顶栏 Tab 对应的六个 `.view`，没想到还有这几个不挂在 `.view` 下面的浮层。已经在原分支上追加一个修复提交把它们纳入同一套深色兜底机制，不重新走一遍 8 个任务的流程。后续阶段二正式迁移这几个浮层所属的功能（用户管理、设置等）时，跟其他页面一样把这层兜底摘掉即可。
