# 素材文案关键词检测 · 设计文档

- **日期**：2026-07-21
- **状态**：脑暴定稿，待写实施计划
- **定位**：新增第六个顶级标签页，独立于价格带沙盘/竞品对位/评论风向标/竞品3D预览/报告管理

## 要解决的问题

运营批量产出素材（主图、详情页图等）时，容易把不属于这个产品的文案/型号错配上去，或者漏放本该出现的必备文案（认证号、核心卖点词等）。人工逐张核对效率低、容易漏。这个功能批量上传素材后自动 OCR 识别图上文字，跟后台配置好的「产品专属关键词」比对，检测出：

1. **串词**：素材里出现了别的产品的专属关键词（型号/文案错配到别的产品图上了）
2. **缺词**：本该出现的这个产品的专属关键词，一个都没在素材里检测到

不做「好评/差评」那种模糊判断，是确定性的字符串包含检测，先测准确度，再决定是否引入容错。

## 技术前提与约束

- 沿用仓库既有的零依赖、无构建步骤、内网离线部署的前提（见 CLAUDE.md）。
- OCR 引擎：**系统级 `tesseract-ocr`（apt 安装 + 中文训练数据），Node 侧用 `child_process` 调用本地二进制**。`install.sh` 里新增一步 apt 安装，跟现在装 Node 是同一套方式，不新增"云端 API 依赖"。
- OCR 执行位置：**服务端**，先跑起来验证识别准确度，后续视实际使用情况（大批量是否卡顿）再评估要不要挪到浏览器端（tesseract.js WASM）。这个决定不是本设计要锁死的，只是 v1 的起点。
- 单台 VM 是单进程 Node，OCR 是 CPU 密集操作，必须有**服务端并发队列**（并发上限，比如 2），避免一次性起太多 tesseract 进程拖垮机器、阻塞其他人正常使用工作台。

## 整体流水线

每张上传的图片：

1. **上传**：原始二进制直传落盘，复用现有上传路径的"不重编码"原则（图片文字清晰度本身也直接影响 OCR 准确率，这点跟竞品对位的"原图不动"诉求是一致的）。
2. **OCR（必然发生）**：不管产品怎么被识别出来，关键词比对都需要这张图的完整文字，所以 OCR 是每张图必跑的一步，不是"识别不出型号才跑"。走服务端并发队列，排队等待时前端显示"识别中..."动画。
3. **产品归属识别（三级，复用同一份关键词数据，不单独维护"识别用标识符"字段）**：
   - **文件名级**：文件名里能唯一匹配到某个产品的名称/型号 → 直接采用。
   - **OCR 反查级**：文件名级不确定时，统计 OCR 文字里命中了哪个产品自己的专属关键词最多；只有一个产品明显领先（≥1 命中且不并列）才采用。
   - **人工选择级**：前两级都无法唯一确定（零命中或并列）→ 前端该行内联显示产品选择下拉框，同时把 OCR 出来的文字摆在旁边给人读——不用重新上传、不用重新跑 OCR（服务端已经把这张图和它的 OCR 结果缓存住，等分辨结果，约 30 分钟未处理则过期丢弃，不写入正式历史记录）。
   - **交叉校验软提示**：即便文件名级已经确定了产品，如果 OCR 文字明显更像属于另一个产品（该产品关键词命中数超过被采用产品），额外标一条琥珀色提示"型号可能填错了"，不影响主流程判定，只是提醒人工复核。
4. **关键词比对**（产品确定之后）：
   - 缺词 = 该产品专属关键词列表里，OCR 文字中找不到的那些（严格子字符串包含匹配，忽略前后空白/换行，不做模糊容错——先测真实 OCR 输出质量再决定要不要加编辑距离阈值）。
   - 串词 = 其它产品的专属关键词，出现在了这张图的 OCR 文字里。
   - 通用词（比如"7天无理由退换"这类哪个产品图上出现都不算问题的词）永远不参与判定，纯粹排除在外。
   - 一个关键词只能属于"某一个产品的专属列表"或"通用词列表"中的一处，不能重复出现在两处——这条唯一性约束在关键词库保存时就校验、拒绝违规保存，运行时匹配逻辑因此不用处理"到底算谁的"这种歧义。
   - 通过 = 缺词和串词都为空；否则不通过。
5. **落库 + 返回**：结果追加写入历史记录（若是人工选择级，等选择完成后才落库），前端渲染这一行的最终状态。

## 数据模型

新增独立数据目录 `materialcheck/`（跟 `reviews/`、`products3d/`、`reports/` 同级，不复用价格带沙盘/3D预览已有的任何产品数据源——那些数据源的所有权、覆盖时机都跟这个功能对不上）：

### `materialcheck/products.json`（管理员维护，整份覆盖式保存，非追加日志）

```json
{
  "products": [
    { "id": "p1", "name": "GC-Multi", "keywords": ["GC-Multi", "抗菌滤网认证号XXX", "静音悬浮马达"] }
  ],
  "universalKeywords": ["7天无理由退换", "包邮", "官方旗舰店"]
}
```

保存时校验：全库所有关键词字符串必须两两不重复（不管是同一产品内部还是跨产品/跨通用词列表），冲突则拒绝保存并指出具体是哪个词、已经属于哪里。

### `materialcheck/records.jsonl`（追加日志，历史记录）

```json
{
  "id": "...", "batchId": "...", "timestamp": "...", "uploadedBy": "username",
  "filename": "original.jpg", "imagePath": "uploads/materialcheck/xxx.jpg",
  "productId": "p1", "productName": "GC-Multi",
  "matchMethod": "filename | ocr | manual",
  "ocrText": "...",
  "missingKeywords": ["..."],
  "crossedKeywords": [{ "keyword": "...", "fromProductId": "p2", "fromProductName": "..." }],
  "status": "pass | fail | ocr_failed",
  "warning": null 
}
```

- `batchId`：同一次批量上传共享一个 id，方便历史记录按"这一批"聚合展示（"12 张 · 通过 9 · 不通过 2 · 待选择 1"）。
- `ocr_failed`：OCR 本身失败/超时（图片损坏、格式异常等），跟 pass/fail 分开计数，不污染合规统计；前端该行给出重试入口。
- 待人工选择产品的图片**不落这份日志**，只在服务端内存/临时文件里缓存（原图 + OCR 文字），超时未处理则丢弃，避免历史记录里堆积"没人处理完"的半成品。

## 后端接口（草案，实施阶段再细化命名）

- `GET /api/materialcheck/products` / `PUT /api/materialcheck/products` — 关键词库读写，仅管理员可写，校验唯一性规则。
- `POST /api/materialcheck/upload` — 单张图片上传 + 走完整流水线（上传→OCR→三级产品识别→比对→落库），同步阻塞到结果返回；若产品识别落到"人工选择级"，返回 `{ needsManualPick: true, pendingId, ocrText, candidateProducts? }` 而不落库。
- `POST /api/materialcheck/resolve` — 传 `{ pendingId, productId }`，用缓存好的 OCR 结果直接做关键词比对并落库，返回最终结果。
- `GET /api/materialcheck/records` — 历史记录查询，支持按产品/状态/上传人/日期范围过滤，按 `batchId` 聚合。

所有人都能调 upload/resolve/records；products 的写接口仅管理员（跟变更日志、备份恢复、用户管理同一套权限模型）。

## 前端 UI

新标签页，页面级三个视图切换（跟报告管理的"第1页/第2页"、评论风向标的"竞品对比/本品分析"是同一种切换模式）：

### 检测台（默认视图）

- 拖拽或点击批量选择图片；前端以有限并发（比如同时 3 个请求）逐个调用上传接口，避免一次性把一堆请求甩给服务端队列。
- 每个文件一行：文件名 → 上传/识别中 spinner（这就是"等待提示动画"，天然由该文件自己的请求 loading 态驱动，不需要额外的全局 loading 状态）→ 完成后变成通过（绿）/不通过（红，可展开看缺词和串词的 chip，串词 chip 标注"属于「产品X」"）/识别失败（灰，带重试按钮）。
- 人工选择级的行内联展示一个产品下拉框 + OCR 文字（供人读图确认），选完即完成该行，不会阻塞其它行的处理。
- "型号可能填错"的软提示以琥珀色小字附在通过状态的行上，不改变整体判定。
- 顶部一行本批次汇总："本次上传 12 张 · 已完成 8 · 处理中 3 · 待选择 1"。

### 历史记录

- 按产品/状态/上传人/日期范围过滤；按批次折叠展开成单张记录列表。
- 点开单条记录看详情：完整 OCR 文字，缺词/串词在文字中做内联高亮（复用评论风向标"悬浮看关键词原句"那套溯源交互的既有模式）。

### 关键词库管理（仅管理员可见，非管理员整个视图连入口都不显示）

- 左侧产品列表（名称 + 关键词数），可增删产品。
- 右侧选中产品的关键词 chip 列表（增删改）+ 独立的通用词 chip 区域。
- 输入的关键词如果已存在于别处，保存前就地报错，提示具体冲突在哪，阻止提交。

### 与现有框架的接入点

- 「功能显示设置」新增第 6 个开关（个人偏好，跟随账号，不影响数据本身）。
- 不参与协同编辑合并（`matrix`/`compare` 才有的机制），因此没有「开启编辑」「撤销/重做」按钮，这两个功能跟评论风向标/3D预览/报告管理一样直接不显示对应按钮。
- 关键词库的增删改建议接入现有 `audit.js` 变更日志（谁改了哪个产品的关键词），跟其它管理操作保持一致的可追溯性。

## 错误处理

- Tesseract 二进制缺失/异常：服务启动时检测，缺失则在日志里明确报错，而不是静默返回空文本导致误判"全部缺词"。
- OCR 单张失败/超时：记为 `ocr_failed`，独立于 pass/fail 统计，前端提供重试。
- 上传格式/大小限制：复用竞品对位现有的图片上传校验（PNG/JPG/WebP）。
- 关键词库唯一性冲突：保存时精确指出冲突的词和已占用它的产品/通用词列表，拒绝保存，不做静默去重或覆盖。

## 测试计划

- 关键词比对逻辑（缺词/串词判定、唯一性校验）是纯函数，不依赖 OCR，可以像 `merge.test.js` 一样写一份零框架的 `materialcheck.test.js`，覆盖：单产品缺词、跨产品串词、通用词豁免、唯一性冲突拒绝保存。
- OCR 准确率本身不写自动化测试（依赖外部二进制和真实素材），按现有"手动冒烟测试"惯例，实施阶段用几张真实素材人工核对识别文字和最终判定是否符合预期，作为决定要不要加模糊匹配容错的依据。

## 明确不做（v1 范围外）

- 不做基于计算机视觉的"自动识别图里是什么产品"（图像分类模型），产品归属完全靠文件名/OCR文字关键词命中 + 人工兜底，不引入训练模型。
- 不做模糊/编辑距离容错匹配，先用严格子字符串匹配观察真实 OCR 输出质量。
- 不做异步任务队列+轮询（本设计的执行模型是同步请求+服务端并发队列），除非实际使用中大批量上传证明同步模型体验不够用。
